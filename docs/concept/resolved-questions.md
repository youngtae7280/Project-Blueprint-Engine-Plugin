# Resolved Questions

This file records questions that are no longer active because decisions or existing public docs answer them.

| ID     | Former Question                                                                                           | Resolution                                                                                                                                                                                                                      | Decision |
| ------ | --------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| RQ-001 | Is PBE a GUI, SaaS backend, daemon, or separate API provider?                                             | No. PBE is a Codex plugin workflow and tree-based development control system.                                                                                                                                                   | DEC-001  |
| RQ-002 | What is the source of product truth?                                                                      | The Product Tree is the source of product truth.                                                                                                                                                                                | DEC-003  |
| RQ-003 | Can Codex implement directly from ambiguous Product nodes?                                                | No. Executable Product nodes need acceptance criteria or an explicit non-executable reason.                                                                                                                                     | DEC-004  |
| RQ-004 | Are Product Tree nodes direct coding tasks?                                                               | No. Project and Work derivation happens through WPD before executable work exists.                                                                                                                                              | DEC-005  |
| RQ-005 | What bounds implementation execution?                                                                     | Cycle Contracts and Node Execution Contracts define selected and foundation scope.                                                                                                                                              | DEC-006  |
| RQ-006 | Who accepts product results?                                                                              | Only the user can accept product results. Codex submits for review.                                                                                                                                                             | DEC-007  |
| RQ-007 | How should product, UX, risk, acceptance, or verification drift be handled?                               | It becomes a Change Node and goes through impact analysis and reopen rules.                                                                                                                                                     | DEC-008  |
| RQ-008 | Are RPD/WPD/VD/ACEP still valid terms?                                                                    | Yes, as compatibility names during the tree-native migration.                                                                                                                                                                   | DEC-010  |
| RQ-009 | Is the old 5 Layer explanation still active architecture?                                                 | No. It is a legacy explanatory frame.                                                                                                                                                                                           | DEC-011  |
| RQ-010 | Does the Graph-source target make Maintainability Graph the source today?                                 | No. Tree-native artifacts remain the current operational source until explicit promotion.                                                                                                                                       | DEC-013  |
| RQ-011 | What exact approval summary belongs in an Approval Brief without weakening the Acceptance Tree rule?      | Approval Brief uses Intent understood, Result summary, Verification summary, Remaining judgment, and Approval choice. It supports Acceptance Tree updates but does not replace Acceptance Tree as durable acceptance state.     | DEC-014  |
| RQ-012 | Are Check and Evidence the same thing?                                                                    | No. Check is the verification obligation; Evidence is observable proof that the Check was performed or satisfied.                                                                                                               | DEC-016  |
| RQ-013 | Can AI self-report count as Evidence?                                                                     | No. AI summary may point to Evidence, but the Evidence itself must be an observable artifact or human judgment record.                                                                                                          | DEC-016  |
| RQ-014 | Is a Control Node executable Work, Evidence, Approval Brief, or durable Acceptance state?                 | No. A Control Node is a separate control record for judgment, change, impact, acceptance closure, and block/reopen status.                                                                                                      | DEC-017  |
| RQ-015 | Should every internal trace be exposed as a Control Node in Approval Brief?                               | No. Only Control Nodes that affect user judgment, warning, blocker, risk/exception/defer acceptance, acceptance closure, or reopen are exposed by default.                                                                      | DEC-017  |
| RQ-016 | Must legacy terms be deleted immediately?                                                                 | No. Legacy and compatibility terms may remain when mapped, marked, and not treated as active authority.                                                                                                                         | DEC-018  |
| RQ-017 | Does Legacy Compatibility Map perform runtime migration or Graph-source promotion?                        | No. It is a concept-level interpretation policy and does not migrate artifacts, change runtime source authority, or promote Maintainability Graph.                                                                              | DEC-018  |
| RQ-018 | Are tree-native artifacts superseded by the compatibility map?                                            | No. Product, Project, Work, Test, Evidence, and Acceptance Trees remain current operational artifacts until a separately approved Graph-source promotion phase.                                                                 | DEC-018  |
| RQ-019 | Where is the boundary between Compatibility Control Node, Legacy Compatibility Map, and superseded items? | Simple term mapping stays in Legacy Compatibility Map or Glossary, replaced directions go to Superseded Items, and mismatches affecting current approval, verification, scope, or user judgment become Control Node candidates. | DEC-018  |
| RQ-020 | Does Runtime Feasibility Demonstration promote Maintainability Graph to current source model?             | No. It is a promotion readiness gate and does not change operational source authority. Tree-native artifacts remain current operational source until a separately approved source transition.                                   | DEC-019  |
| RQ-021 | Can AI self-report alone count as runtime feasibility Evidence?                                           | No. Representative feasibility requires observable Evidence and linked records. AI summaries may point to Evidence but cannot replace it.                                                                                       | DEC-019  |
| RQ-022 | Must a runtime feasibility demo implement every runtime feature before promotion can be considered?       | No. It must cover a representative lifecycle slice with the required happy path, stale/reopen, decision, evidence exception, compatibility mismatch, and scope boundary scenarios.                                              | DEC-019  |
| RQ-023 | Is Source Transition Path the same as Graph-source promotion?                                             | No. It is a concept-level authority transition policy and prerequisite path. It does not move source authority or declare promotion.                                                                                            | DEC-020  |
| RQ-024 | Does Graph read-model alignment change source authority?                                                  | No. Graph read-model alignment explains and cross-checks tree-native artifacts, but tree-native artifacts remain current operational source until explicit promotion.                                                           | DEC-020  |
| RQ-025 | After promotion, can Codex/PBE replace user acceptance authority?                                         | No. User-controlled acceptance authority remains required before and after any source-model promotion.                                                                                                                          | DEC-020  |
| RQ-026 | Should Source Transition Path be a separate concept policy?                                               | Yes. Source authority transition needs a dedicated policy boundary before rollback/compatibility strategy or implementation details are designed.                                                                               | DEC-020  |
